O Problema

O desenvolvimento artesanal
é o gargalo da inovação.

Ideias morrem no vale entre o conceito e o código.

A consequência? Empresas perdem tempo corrigindo o que a IA "tentou" fazer.
  • Squads inflados: meses para validar uma hipótese simples.
  • A IA alucina: inventa soluções que não funcionam.
  • Falta de padrão: cada entrega vem de um jeito diferente.
  • Insegurança: risco de quebrar o que já estava pronto.

A Solução

Da conversa ao MVP em 48h.
Com contratos, não com sorte.

Entrada

Briefing

60 min de conversa gravada e transcrita.

Processo

Pipeline Governado

11 camadas de validação automática entre o briefing e o build.

Saída

MVP no ar

Software funcional entregue em 48h de build.

Blueprint-First
Contrato antes do código

Nenhuma linha de código sem contrato técnico aprovado.

Determinístico
Mesmo input, mesmo output — sempre

Cada etapa depende da anterior. Previsível por design.

Anti-hallucination
Cada agente opera dentro de limites rígidos

Zero improviso. O contrato impede que a IA invente.

03 / 20

Arquitetura

Três camadas.
Cada uma especializada, auditável e independente.

01
Camada 1 — Thinker
"Moedor de Caos"

Transforma transcript.txt em intenção técnica estruturada com scoring de features, story points e priorização por valor de negócio.

transcript.txt thinker_output.json thinker_spec.html
02
Camada 2 — Core
"Motor de Governança"

11 camadas de validação automática. Normaliza a intenção (Bridge), compila a arquitetura (Architect), valida coerência e produz o plano de obra final.

refined_intent.json product_definition.json builder_handoff.json
03
Camada 3 — Builder
"Mestre de Obras"

Claude Code lê o builder_handoff.json e materializa o MVP. Nunca toca protected_zones. Só escreve nos injection_slots definidos.

Next.js + Tailwind v4 shadcn/ui MVP funcional

Os Protocolos .md

O sistema nervoso.
Cada agente sabe exatamente o que fazer.

Os protocolos .md são arquivos Markdown que definem o comportamento, as regras e os limites de cada etapa — o "cérebro" de cada agente.

core/ — estrutura de protocolos 
core/
├── product-thinking-agent/      ← Camada 1
│   ├── thinker.md              ← O "cerebro" do Thinker
│   ├── briefing-steering.md    ← Roteiro da entrevista
│   ├── manifest.md             ← Filosofia da camada
│   └── product-thinking-operation.md
│
├── architect-agent/             ← Camada 2a
│   └── arquitect.md            ← Regras do Architect (LLM)
│
├── builder-agent/               ← Camada 3
│   └── builder.md              ← Step-by-step do Builder
│
├── design-grammar/              ← Regras visuais
│   └── README.md               ← Tokens, layouts, proibições
│
└── skeletons/                   ← Contrato do esqueleto
    ├── skeleton_manifest.json  ← Capabilities e resolução
    └── nextjs_app_base.md     ← Documentação do template

06 / 20

Protocolo

thinker.md
O Moedor de Caos.

thinker.md Lido por: thinker/main.py (FastAPI /moer)
O que faz
  • Filtra o ruído da conversa
  • Extrai o fluxo crítico (Happy Path A→Z)
  • Mapeia entidades e relacionamentos
  • Define Must-have vs Nice-to-have
  • Calcula impact_score (1-10) e story_points (Fibonacci)

Output

thinker_output.json 
{
  "project_name": "PetFlow",
  "archetype": "dashboard",
  "features": [
    {
      "id": "feat_01",
      "title": "Cadastro de Pet",
      "impact_score": 9,
      "story_points": 3,
      "priority": "P0"
    }
  ],
  "sprint_capacity": 40
}

07 / 20

Protocolo

arquitect.md
O Engenheiro Chefe.

arquitect.md Lido por: self-inferred-context.py (Claude API)
O que faz
  • Escolhe archetype e navigation_pattern
  • Define core_entities e primary_flows
  • Mapeia páginas com rotas e componentes
  • Aplica degradation_policy quando necessário
Regras inegociáveis
  • Nunca inventar infraestrutura fora do intent
  • Se database.enabled: false → sem banco de dados
  • Se realtime desabilitado → sem websocket

Output

product_definition.json
Blueprint técnico completo

Archetype, páginas, rotas, componentes, features, design grammar — validados por JSON Schema antes de avançar.

08 / 20

Protocolo

builder.md
O Mestre de Obras.

builder.md Lido por: Claude Code
O que faz
  • Materializa o skeleton em clients/<project_slug>/
  • Cria páginas nos injection_slots
  • Nunca toca protected_zones
  • Instala apenas libs das capabilities

Checklist — Definition of Done

  • Autenticação em todas as rotas
  • Todo button com onClick ou type="submit"
  • Nenhum HEX em .tsx
  • Nenhum any em TypeScript
  • npm run build sem erros

Quick Reference

Glossário

Termo Significado
Blueprint product_definition.json — planta técnica do produto
Handoff builder_handoff.json — plano de obra para o Builder
Skeleton Template base Next.js + Tailwind (nextjs_app_base)
Injection Slots Pastas onde o Builder pode escrever código
Protected Zones Arquivos que o Builder JAMAIS modifica
Capability Feature técnica: auth, persistence, analytics, etc.
Degradation Quando uma feature excede capabilities → fallback documentado
Fail-Fast Pipeline para imediatamente se algo falha
